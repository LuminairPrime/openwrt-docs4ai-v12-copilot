# AI Evaluation and Scoring Framework

This document outlines the formal methodology for mathematically scoring the outputs generated by the AI Defect Discovery Pipeline. It defines how we translate raw code snippets and auditor tags into a quantitative **OpenWrt Competence Score (OCS)**, and how we use those scores to dictate the documentation pipeline.

---

## 1. The Grading Rubric (Binary Pass/Fail)

When an AI generates a snippet for a scenario, it is graded on a strict binary Pass/Fail scale. Because the scenarios represent fundamental, minimal boilerplate, any architectural mistake results in an immediate failure for that snippet.

*   **[ 1 ] Pass (Flawless Execution)**
    *   *Definition:* Perfect, modern OpenWrt code natively utilizing `uci`, `ubus`, `uloop`, and `procd` where appropriate. No legacy APIs or generic Linux hallucinations are present.
*   **[ 0 ] Fail (Architectural Defect)**
    *   *Definition:* The snippet contains one or more errors. This includes catastrophic hallucinations (e.g., writing an Ubuntu `systemd` unit instead of `procd`), legacy approaches (e.g., using Lua CBI instead of JS), or functional code missing mandatory OpenWrt boilerplate (e.g., forgetting `#include <libubus.h>`).
    *   *Action:* The Evaluating Agent MUST assign one or more tags from the Failure Taxonomy to explicitly categorize *why* the snippet failed.

---

## 2. The Failure Taxonomy

Every time a scenario scores a `0`, the Evaluating Agent MUST assign one or more tags from our taxonomy library to explicitly categorize the failure.

*   `ERR_LINUX_HALLUCINATION`: Used standard Linux utilities (systemd, bashrc, ip).
*   `ERR_LEGACY_API`: Used deprecated OpenWrt patterns (Lua, legacy netifd calls).
*   `ERR_MISSING_BOILERPLATE`: Forgot mandatory structure (USE_PROCD, uloop init).
*   `ERR_STATE_MANAGEMENT`: Used flat text files in `/etc/` rather than the `uci` system.
*   `ERR_NON_C_COMPLIANT`: Missed internal C architectural standards (like `blobmsg` dictionaries).

---

## 3. The OpenWrt Competence Score (OCS) Matrix

By running all 17 scenarios from our test catalog, we extract a **Competence Matrix** for any given AI model. We group the scores by their language domains to find out *exactly* what the AI is bad at.

---

## 4. The Proposal Workflow (Closing the Loop)

The entire purpose of this pipeline is to stop guessing what documentation is needed and start building exactly what is missing.

1.  **Analyze the OCS Matrix:** In the example above, the AI scored **0%** in LuCI JS and **16%** in uCode. It is fundamentally ignorant of OpenWrt's modern web and scripting layers.
2.  **Generate the Proposal:** Based on the scores, the project manager proposes two immediate deliverables:
    *   *Cookbook Chapter 1:* "Modern LuCI UI: Views and Maps"
    *   *Cookbook Chapter 2:* "uCode 101: Interacting with the System Bus natively."
3.  **Execute & Retest:** We author those two markdown files. We load them into the AI's context window (our RAG pipeline). We run the 17 scenarios again.
4.  **ROI Measurement:** If the OCS in LuCI JS jumps from 0% to 100%, we mathematically prove that our specific piece of documentation perfectly solved the AI's hallucination.
