## 2. Prompt Quality Assessment — `05-v13-development-prompt-2026-03-22.md`

**Rating: 9/10**

This is a genuinely excellent orchestration prompt. Most prompts for multi-phase work at this scale either over-specify (locking the agent into bad decisions) or under-specify (leaving the agent to invent structure). This one hits the right balance.

### What worked extremely well

**The global rules section is the most valuable part of the document.** Rules like "fix root causes, not symptoms" and "use skeletal authored cookbook files as live pipeline fixtures before writing heavy content" are the kind of constraints that prevent an hour of competent implementation from going in the wrong direction. Rule 10 ("When a plan file specifies an exact field name, use that exact specification — don't silently deviate") is exactly the right discipline for a project with a tight metadata contract.

**The hard guard rails section is unusually good.** Pinning the A8 exclusion seed entries by name, banning `llms-mini.txt`/`llms-small.txt` explicitly, and scoping dead-link checks to release-tree-only are all decisions that would otherwise generate costly re-work if left implicit. Every one of those rails was hit at least once during implementation — the rails did their job.

**The conflict resolution section anticipates a real problem.** Three plan files plus a baseline audit plus a development prompt creates a non-trivial reconciliation surface. Having a clear priority order (live code > later plans > acceptance tests, and this prompt > plan files on guard rails) prevented ambiguity on at least three occasions.

**Phase ordering is correct.** The dependency graph — docs scaffold → metadata → cookbook ingest → pipeline wiring → AGENTS → routing → tests — is the right order and the prompt respects it.

### Minor criticisms

**Phase 8 was underspecified relative to what it touched.** The checklist says "fix root routing" but the actual implementation required: (a) a new `MODULE_DESCRIPTIONS` dict in `06`, (b) a new `CATEGORY_ORDER` entry, (c) an updated `release-inputs/release-include/README.md`, AND (d) fixing a previously hidden test assumption about what `./llms.txt` etc. links look like in the overlay README. The prompt and plans treat Phase 8 as a single bullet but it's actually four interdependent pieces. The test failure that surfaced during Phase 8 was a direct result of this gap.

**"Verify overlay copy order" in Phase 7 is the weakest bullet.** The actual verification required reading `07`'s code, checking the workflow stage order, and understanding that `apply_release_include_overlay` runs after `05b` — none of which is trivially discoverable from the prompt alone. A one-sentence explanation of _why_ this would be at risk (and what to look for) would have made this faster.

**The "deferred phases" guidance** ("Phases 12 and 13 may safely run in a follow-up session") is correct but could have added "Phases 10 and 11 are also safe to defer" to the same sentence. As written, an agent reading only the prompt might infer that Phases 10–11 must complete in the same session, creating unnecessary pressure.

## 4. Implementation Quality Assessment

### What went well

**The metadata provenance phase (Phase 3) was the highest-risk and it went cleanly.** Eight `02*` scripts with different scraping logic needed the same structural changes. By implementing `lib/source_provenance.py` first with a consistent API and then applying it uniformly, the changes were rote rather than creative. No test failures during Phase 3.

**Phase 5 (cookbook ingest) was a model phase.** The spec was precise, the implementation was a single new file of ~100 lines, the fixture test was immediate, and the fail-fast behavior was verified concretely. This is what "small, coherent edits with validation after each phase" looks like when it works.

**Phase 9 surfaced the right issue.** The inline dead-link block in `validate()` was scoped to `release-tree/` but not callable independently and thus not testable. Extracting it to `check_dead_links(release_tree_dir, hard_fail)` and calling it from inside `validate_release_tree_contract()` was the right refactor — it made the function testable and moved the check to the right conceptual level. Four unit tests now cover behavior that was previously only tested through the full pipeline.

**The working checklist's assumption recording paid off.** The `A8` assumption about `original_url → source_url` renaming prevented a question that would have required a context-expensive read. The `A5` assumption about dead-link scope prevented over-engineering (no need to validate L1/L2 cross-links, which don't resolve at the source path level).

### Where there was friction

**Phase 8 generated 79/80 test failure** before the README markdown links were corrected. The root cause was that the smoke test checked for literal `./llms.txt`, `./AGENTS.md`, `./ucode/map.md` strings in the overlay README — but the new README was written without `./`-prefixed markdown links. This was a non-obvious coupling between the test fixture (`pytest_02`) and the overlay content. The fix was correct (proper markdown links) but discovering the test assertion required reading test code under failure conditions. Better: the test could have rendered a clearer failure message explaining what exact string it was checking for.

**The workflow YAML edit for `02i` required reading the surrounding job context** before making the change. The workflow uses a specific parallel fan-out pattern that isn't obvious from the script names. This wasn't a planning gap — it was just an inherently context-heavy edit.

**Phase 2 (docs-new) was the highest word volume with the lowest validation signal.** Creating five documentation files is the kind of work where it's easy to produce something that looks correct but has subtle inconsistencies with the actual codebase. The "verify no broken internal links between docs-new files" checklist item was well-intentioned but difficult to verify without a link-checking pass. These docs should be audited against the actual codebase state in a future session.

## 5. Lessons for Future Development Cycles

### Lesson 1: Keep the baseline audit as a document type

`01-v13-baseline-verification-audit` is the highest-quality document in the set, and it should become a standard artifact in every release planning cycle. The pattern is: write a tier list (00), verify the tier list's claims against live code (01), then write the implementation specs (02+). This sequence prevents spec work from resting on unverified assumptions.

### Lesson 2: The assumption log in the working checklist is load-bearing

The ten assumptions recorded at Phase 0 were all returned to. `A5` (dead-link scope), `A8` (wiki URL rename), and `A2` (`source_commit` conditionality) each saved at least one context-expensive rediscovery. The format — a table with assumption, basis, and the specific plan reference that triggered it — is worth standardizing.

**Recommendation:** Make assumption recording explicit in future development prompts. Add a line like: "Before starting Phase N, read all assumptions in the working checklist and flag any that Phase N's changes might invalidate."

### Lesson 3: Multi-phase prompts need per-phase output contracts

The development prompt specifies tasks per phase but doesn't always specify the exact observable output that marks a phase as done. "Run `python tests/run_pytest.py`" is a good completion signal. "Verify overlay copy order" is not — it lacks an observable outcome. The best phases in this session were the ones where the completion signal was unambiguous (test pass count, concrete file creations, fail-fast behavior verified).

**Recommendation:** For every checklist item, write the observable completion criterion next to it. Not "implement X" but "implement X — verified by Y."

### Lesson 4: Test assertions that rely on exact string matching in generated files need documentation

The Phase 8 failure happened because a test checked for `./llms.txt` in the output README, but this assertion was not documented in either the spec or near the test. Anyone writing the overlay README without having read that specific test would produce the same failure. This is a hidden coupling.

**Recommendation:** When a test asserts exact string content in a generated or overlay file, add a comment in the overlay file itself referencing the test. Even a one-line comment catches this before it happens.

### Lesson 5: Documentation phases need validation that code phases don't

Phase 2 created five `.md` files in `docs/docs-new/`. There is no automated test that these files accurately describe the codebase. They can become stale silently. The "no broken links" check at the end of Phase 2 catches structural issues but not semantic drift.

**Recommendation:** For documentation heavy phases, add one automated smoke test that checks key claims. For `pipeline-stage-catalog.md`, a test that verifies every `.github/scripts/*.py` filename is mentioned in the catalog would prevent slow drift. This is a small test with high signal value.

### Lesson 6: Overlay files and static includes need index discovery

The per-module `AGENTS.md` files in `release-inputs/release-include/{module}/` cannot be discovered by an agent that doesn't already know they exist. There is no manifest, no glob, no test that enumerates them. They are copied correctly — the overlay mechanism works — but the overlay directory has no self-documenting structure.

**Recommendation:** Add a test that enumerates files under `release-inputs/release-include/` and confirms each one appears in the final release-tree after a fixture pipeline run. This makes it harder to create a file that silently doesn't ship.

### Lesson 7: Phase boundaries should be decided by refactor risk, not by feature chunking

Phases 3 and 6 were correctly scoped: each made many small changes across many files, all towards one coherent goal. Phase 8 felt mis-scoped: it bundled `MODULE_DESCRIPTIONS` dict (a two-line code change), `CATEGORY_ORDER` update (a one-line change), README expansion (a sixty-line authoring task), and the discovery of a hidden test dependency — all under one phase header. The test failure wasn't risky, but if Phase 8 had involved more complex logic, bundling into a single phase would have made failure diagnosis harder.

**Recommendation:** When a phase mixes code changes with authoring/content work, split them if possible. Code changes have deterministic validation signals. Authoring work is harder to validate automatically and creates noise when debugging unrelated failures.

### Lesson 8: The "debrief" document type should be standard

Writing this document is forcing a structured review of what actually happened vs. what the plan expected. The test failure in Phase 8, the friction points in Phase 2, and the refactoring insight from Phase 9 all belong in institutional memory. Without a debrief, the next development cycle starts from the plans without knowing which friction points are inherent to the codebase and which are avoidable with better prompting.

**Recommendation:** Make `NN-v{X}-debrief-{date}.md` a required deliverable for every release. Write it last, in a separate invocation, specifically because writing it last means having the sharpest possible recall of what actually happened.

Before committing, read 2–3 `js_source-api-*.md` files to assess whether the JSDoc extraction in `02c` produces enough structural signal to support type inference, or whether hand-authored `.d.ts` stubs are the right approach.

## 7. Overall Assessment

The v13 implementation, through Phase 9, went well. The planning documents were accurate, the phase ordering was correct, and the codebase proved more amenable to the changes than expected (pipeline auto-discovery meant no structural rewiring was needed for cookbook). The test suite grew from 80 to 84 tests with no regressions.

The unfinished work (Phases 10–13) is well-scoped and deferrable. None of the completed phases left the system in a state that blocks the deferred ones. The metadata contract (Phase 3) and the cookbook pipeline wiring (Phases 4–6) are the prerequisites for everything else, and those are done.

The main risk in the deferred phases is Phase 11 (A6 visible provenance) touching assembled output — every chunked reference file will gain a provenance header, which is a large surface change. That phase should be validated particularly carefully against the release-tree contract tests.

**Verdict:** The planning and prompt quality for v13 were unusually good. The main gap is that some phases mixed code and content work in ways that made failure diagnosis noisier than necessary. The debrief lesson about observable completion criteria is the highest-leverage change for future cycles.
