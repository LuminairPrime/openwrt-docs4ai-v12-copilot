# AI Agent Prompt Pack v2: Python Pipeline Testing

## Purpose

Reusable, concise prompts for AI coding agents to analyze, test, and harden Python file-processing pipelines. Designed for pipelines that extract data from external sources, transform and normalize it, validate it, and write structured output artifacts.

Tested against: documentation-generation pipelines with multi-stage extract → normalize → assemble → validate → deploy workflows running in CI/CD.

---

## Glossary

| Term | Definition |
|---|---|
| **Record** | A single unit of extracted data (one wiki page, one API module, one package) |
| **Stage** | A pipeline phase: extract → normalize → assemble → validate → output |
| **Artifact** | Any file the pipeline writes (markdown, JSON, HTML, YAML, `.d.ts`) |
| **False success** | Pipeline exits 0 and writes output, but output is wrong, incomplete, or corrupted |
| **Slug** | A URL/filesystem-safe identifier derived from a title or path |
| **Skip flag** | An env var or CLI flag that disables a pipeline stage |
| **Frontmatter** | YAML metadata block at the top of a markdown file between `---` delimiters |
| **L1/L2/L3/L4** | Pipeline layer levels: raw → semantic → navigational → monolithic |

## How to Use

### Prerequisites

```
pip install pytest hypothesis pyyaml
```

Optional, for mutation testing:

```
pip install mutmut
```

### Context Template

Include this header with your code when giving any prompt to an AI agent:

```
Pipeline: [1-2 sentence description of what the pipeline does]
Scope: [all code / specific files / specific stages]
Existing tests: [none / basic / comprehensive]
Known issues: [list, or "none known"]
```

### Recommended Workflows

**Quick audit (1-2 hours):** Prompts 1, 2, 5 → pipeline map, core bugs, false-success risks.

**Full static analysis (half day):** Prompts 1-5, then prompt 6 for unit tests, then prompt 14 to review generated tests.

**Comprehensive hardening (1-2 days):** All prompts in order. Finish with prompt 15 to consolidate.

### Prompt Dependencies

- **Prompt 9** (Mutation Testing) requires existing tests — run 6-8 first if none exist.
- **Prompt 10** (Differential Testing) requires two comparable versions or modes to compare.
- **Prompt 14** (Test Quality Review) reviews tests generated by prompts 6-8.
- **Prompt 15** (Triage) aggregates findings from all previous prompts.

### Multi-Agent Comparison

Give the same prompt and identical code to multiple agents. Compare finding counts, severity distributions, and overlap. Trust findings reported by multiple agents. Use prompt 15 to merge results.

## Shared Rules

These apply to every prompt. Copy them alongside any single prompt when used in isolation:

1. **Cite exact code.** Every finding must reference a specific function name, variable, or quoted snippet.
2. **No fabrication.** Do not invent functions, variables, files, or line numbers absent from the provided code.
3. **Mark confidence:** `confirmed` (provable from code), `likely` (strong evidence), `speculative` (uncertain).
4. **Severity scale:**
   - **Critical** — silent data corruption, security vulnerability, data loss
   - **High** — wrong results, partial failure reported as success, broken output
   - **Medium** — edge-case failures, poor error handling, missing validation
   - **Low** — maintainability, clarity, minor robustness gaps
5. **No padding.** "No issues found" is valid and preferred over fabricated findings.

## 1 · Pipeline Map & Codebase Inventory

> **When to use:** Always. Run first to establish shared understanding.
> **When to skip:** Never.
> **Tools:** None.

```text
You are reviewing a Python file-processing pipeline. Before any bug analysis, map the codebase accurately.

Tasks:
1. List every script in execution order with a one-line purpose summary.
2. For each script, identify: inputs (files, env vars, network), outputs (files written, side effects), external tool dependencies (git, pandoc, npm, etc.), and skip/abort conditions.
3. Map shared library modules — what they export and which scripts import them.
4. List every environment variable that alters behavior.
5. List all output artifacts and expected locations.
6. Flag anything that looks incomplete, broken, contradictory, or dead — but mark as observations, not deep analysis.

Output format:
- Script table: | Script | Purpose | Inputs | Outputs | External Tools | Skip Conditions |
- Env-var table: | Variable | Used By | Effect |
- Shared module interface summary
- Artifact inventory
- Preliminary red flags with confidence (confirmed / likely / speculative)

Do not deeply analyze bugs — later prompts do that. Focus on accurate, complete mapping. If code appears truncated, malformed, or contains duplicate blocks, note it explicitly.

Rules: Cite exact code. No fabrication. Mark confidence.
```

## 2 · Static Analysis — Correctness & Dataflow

> **When to use:** Always. Primary bug-finding prompt.
> **When to skip:** Never.
> **Tools:** None (pure reasoning).

```text
You are performing a static correctness analysis of a Python file-processing pipeline. Do not execute the code.

Goal: Find bugs, broken logic, type errors, and paths where output can be silently wrong.

Focus areas:
- Dataflow: trace data from input through every stage to output. Find where chains break.
- Function contracts: infer what each function expects and returns. Flag callers that pass wrong arguments. Flag functions whose bodies contradict their names or docstrings.
- None/empty handling: missing checks on None, empty collections, missing dict keys, blank strings.
- Variable scoping: variables used before assignment, referenced from the wrong scope, conditionally defined but unconditionally used.
- Schema assumptions: code assuming dict keys, JSON structure, or file content exist without checking.
- Error masking: bare except blocks, swallowed return values, fallback logic that hides failures.
- Output integrity: paths where incomplete or corrupted output is written while the script exits 0.

Instructions:
1. Read all code first. Build a data-structure model at each stage.
2. For each finding: severity, confidence, exact location (quote the code), why it's risky, concrete failure scenario, minimal fix.
3. After all findings: top 5 highest-risk bugs, assertions to add between stages, functions to refactor.

Trace backward from every file-write call and verify all data reaching it was validated.

Be suspicious of:
- Functions whose body implements a different function's logic
- Duplicate import blocks or repeated initialization code
- Variables referenced but never defined in the enclosing scope
- Content generated as a string but never written to disk
- Extract/scrape failures that produce empty but "successful" outputs

Rules: Cite exact code. No fabrication. Mark confidence. No style nits.
```

## 3 · Static Analysis — Security & Robustness

> **When to use:** Always for pipelines processing external data.
> **When to skip:** Internal-only pipelines with fully trusted inputs (rare).
> **Tools:** None.

```text
You are performing a static security and robustness analysis of a Python pipeline that processes external data (git repos, web pages, APIs, filesystem).

Goal: Find trust-boundary violations, injection risks, and denial-of-service vectors.

Focus areas:
- Trust boundaries: where untrusted data enters and where it is used without sanitization.
- Path safety: filenames or paths derived from external data, directory traversal, unsafe joins.
- Subprocess safety: commands built with string formatting or user-controlled arguments, shell=True usage.
- Network hygiene: missing timeouts, unchecked HTTP status codes, unbounded response sizes, retry storms.
- Deserialization: JSON/YAML loaded without schema validation, bare except around parsing.
- Resource exhaustion: unbounded loops, giant string concatenation, entire files read into memory without size limits.
- Content injection: scraped content that could inject YAML frontmatter delimiters, markdown code fences, or structural elements into output.
- Secret exposure: API tokens or credentials logged on error paths or embedded in output.
- Regex safety: patterns vulnerable to catastrophic backtracking on adversarial input.

Instructions:
1. Map all external inputs and output sinks.
2. For each finding: severity, confidence, exact location, realistic attack/failure scenario, safer replacement.
3. Provide a prioritized hardening checklist.

Be suspicious of:
- Slug/filename construction from scraped text without length or character limits
- Subprocess calls passing external data as arguments
- Scraped HTML/wiki content written into markdown without sanitization
- Cache files loaded without integrity or version checks
- API tokens accessible via environment variables and logged on failure

Output: trust-boundary map, findings table, hardening checklist, safer patterns.

Rules: Cite exact code. No fabrication. Mark confidence.
```

## 4 · Static Analysis — Config / Schema / Flag Contracts

> **When to use:** Pipelines with environment variables, skip flags, multi-mode behavior, or multi-stage schemas.
> **When to skip:** Trivial single-file scripts with no configuration.
> **Tools:** None.

```text
You are auditing configuration, schema contracts, and mode-flag interactions across a multi-script Python pipeline.

Goal: Find mismatches between environment variables, skip flags, schema assumptions, and output expectations across pipeline stages.

Focus areas:
- Config drift: env vars read inconsistently, duplicated, or with conflicting defaults across scripts.
- Skip-flag interactions: flag combinations that produce empty output, skip validation, or leave downstream stages expecting missing artifacts.
- Schema contracts: record shape at extraction vs normalization vs assembly vs output. Fields assumed mandatory in one stage but optional or absent in another.
- Default-value hazards: None defaults, empty strings, or magic constants that silently change behavior when a key is missing.
- Cross-stage interfaces: functions or scripts returning different structures depending on branch, error case, or mode.
- Cache contracts: cached data loaded without schema or version validation.
- Output format consistency: whether all code paths produce the same required fields and file structure.

Instructions:
1. Build an interface map: env vars, skip flags, per-stage input/output schemas, artifact contracts.
2. Identify dangerous flag combinations (e.g., all extraction skipped but validation still expects output).
3. For each finding: severity, confidence, exact location, failure scenario, fix.
4. Propose: canonical config model, stage-boundary validators, illegal flag combinations to reject at startup.

Be suspicious of:
- Flags that disable stages without adjusting downstream expectations
- Config values interpreted as different types in different scripts
- Shared constants defined in multiple places with different values
- Cache files assumed to match a schema that may have changed

Output: interface map, findings table, proposed config model, startup validation rules.

Rules: Cite exact code. No fabrication. Mark confidence.
```

## 5 · Static Analysis — Dead Code & False-Success Paths

> **When to use:** Always. Catches the most dangerous production failures.
> **When to skip:** Never.
> **Tools:** None.

```text
You are auditing a Python pipeline for dead code, unreachable branches, and false-success paths.

Goal: Find code that cannot execute, conditions that are always true/false, and paths where the pipeline reports success while producing wrong or incomplete output.

Focus areas:
- Dead code: functions, branches, handlers, or validation logic that can never execute.
- Unreachable branches: contradictory conditions, shadowed branches, logic bypassed by earlier returns.
- False-success paths: scripts that exit 0 or write output even when extraction, parsing, or validation failed.
- Dead validation: validators whose results are ignored, warnings that should be fatal, checks that log but don't gate output.
- Contradictory success logic: "completed" vs "non-empty" vs "file written" treated as interchangeable when they aren't.
- Partial-state hazards: temp files, caches, or output directories left in a misleading state after failure.
- Count mismatches: counters logged from pre-filter data while files are written from post-filter data.

Instructions:
1. Sketch the control-flow graph. Identify all success, partial-success, and failure exits.
2. For each finding: severity, confidence, exact location, failure scenario, fix.
3. After findings: list branches to delete, success conditions to tighten, output-integrity assertions to add before file writes.

Trace every sys.exit(0) and implicit-success return. Verify the pipeline has actually produced valid, complete output before claiming success.

Be suspicious of:
- "Success = no exception" even when nothing was produced
- Empty outputs treated as valid
- Scripts that print "[XX] FAIL: ..." but continue instead of exiting
- Write operations that happen before validation completes
- Multiple code paths to "success" with inconsistent quality checks

Output: control-flow exit map, findings table, assertions to add, dead code to remove.

Rules: Cite exact code. No fabrication. Mark confidence.
```

## 6 · Unit Test Generation

> **When to use:** Always. Foundational tests for any codebase.
> **When to skip:** If comprehensive unit tests already exist.
> **Tools:** pytest.

```text
You are writing pytest unit tests for a Python file-processing pipeline.

Goal: Generate executable tests for individual functions, covering normal operation and key error cases. Tests must run without network access, filesystem side effects, or external tools.

Instructions:
1. Identify all testable functions: parsers, normalizers, slug generators, metadata extractors, validators, formatters, content transformers.
2. For each function, write tests covering:
   - Normal input → expected output
   - Empty input (empty string, empty list, empty dict, None where applicable)
   - Missing keys or fields
   - Boundary values (very long strings, single characters, edge-length slugs)
   - Malformed input that should be handled gracefully
3. Mock or stub: file I/O, network calls, subprocess calls, environment variables, datetime.now().
4. Use pytest fixtures for shared setup. One assertion concept per test.
5. Name tests descriptively: `test_<function>_<scenario>`.

Requirements:
- Runnable with `pytest` alone (plus pip-installable mocks).
- No calls to external tools (pandoc, jsdoc2md, git, npm).
- No network requests. No real filesystem writes.
- Test functions in isolation, not whole scripts.
- Include at least one test per function that verifies empty/None input behavior.

Priority targets:
- Slug and filename generation
- Metadata extraction from file content
- Content sanitization and transformation
- YAML frontmatter injection and parsing
- Validation checks and gating logic
- Cross-link registry symbol extraction

Output:
- Test file(s) with imports, fixtures, and test functions
- Brief note per test explaining what bug class it guards against

Rules: Only test functions that exist in the provided code. No fabricated function names.
```

## 7 · Property-Based & Metamorphic Testing

> **When to use:** After unit tests exist. Best for normalization, deduplication, serialization, and identity logic.
> **When to skip:** Trivial pipelines with no normalization or deduplication.
> **Tools:** pytest, hypothesis (`pip install hypothesis`).

```text
You are writing property-based tests using pytest and Hypothesis for a Python pipeline that extracts, normalizes, and outputs structured data.

Goal: Find silent logic bugs where the pipeline succeeds but output is wrong, inconsistent, or unstable.

Tasks:
1. Identify key invariants at each stage.
2. Build Hypothesis strategies for records, metadata dicts, file content, and slugs.
3. Write property tests asserting invariants and metamorphic relations.

Properties to test:
- Schema integrity: every output record has all required fields with correct types.
- Idempotence: normalizing a record twice produces the same result as once.
- Round-trip stability: parse → serialize → parse preserves required fields.
- Slug determinism: same input always produces same slug.
- Slug safety: output slugs are always valid filesystem names (no empty, no path separators, no reserved characters).
- Batch isolation: one malformed record does not corrupt unrelated records.
- Encoding robustness: equivalent Unicode forms and whitespace variants do not create false duplicates.

Metamorphic relations to test:
- Adding trailing whitespace/newlines to content must not change record identity or slug.
- Reordering input records must not change output content (if pipeline claims order independence).
- Processing N records as one batch vs two half-batches must produce equivalent merged results (if pipeline claims batch independence).

Implementation:
- Use `@given` decorators with composable strategies.
- Shrink failing cases to minimal examples.
- Target parse, normalize, and slug-generation functions directly.
- Flag silent output corruption as test failure even without exceptions.
- Save minimal failing examples as regression fixtures.

Output:
- Hypothesis strategies for pipeline data types
- Property test code with docstrings explaining each property
- Notes on which properties are highest risk

Rules: Cite exact functions under test. No fabricated APIs.
```

## 8 · Edge-Case Stress Testing

> **When to use:** After basic unit tests exist. Essential for parsers processing external data.
> **When to skip:** Pipelines with fully controlled, schema-validated input.
> **Tools:** pytest.

```text
You are writing pytest tests that feed adversarial and boundary-case inputs into a Python pipeline's parser and transform functions.

Goal: Find crashes, hangs, resource exhaustion, and malformed output caused by unusual but plausible inputs. No fuzzing framework required — use pytest.parametrize with hardcoded edge cases.

Input categories to cover:
- Encoding: UTF-8 BOM, replacement characters, zero-width characters, mixed encodings.
- Whitespace: tabs, CRLF vs LF, bare CR, trailing whitespace, lines of only spaces, blank documents.
- Size: empty string, single character, 1 MB string, content with 50K+ lines.
- Malformed structure: unclosed HTML tags, nested code fences, YAML-like `---` inside body text, JSON with trailing commas or duplicate keys.
- Markdown edge cases: headings inside code blocks, backticks in headings, links containing parentheses.
- Slug/path hazards: input names with unicode, dots, double hyphens, leading/trailing hyphens, path separators, empty names after sanitization.
- Timestamps: None, empty string, pre-epoch dates, far-future dates, timezone-ambiguous formats.
- Hostile content: HTML error-page signatures embedded in valid content, embedded `<script>` tags, YAML injection via `---` in scraped text.

Instructions:
1. Identify the pipeline's parser and transform entry points.
2. Write parametrized tests organized by category.
3. Assert: no unhandled exceptions, no hangs (use `@pytest.mark.timeout`), no empty output on valid input, no corrupt output structure.
4. Keep tests deterministic, fast, and independent of network/filesystem.
5. Clearly separate "expected rejection" tests from "should succeed" tests.

Output:
- Parametrized test code by category
- Timeout markers for hang-risk tests
- Notes on which inputs exposed real issues vs expected rejections

Rules: Only test functions in the provided code. No fabricated targets.
```

## 9 · Mutation Testing Analysis

> **When to use:** After prompts 6-8 have produced tests. Evaluates test-suite quality.
> **When to skip:** No tests exist yet (run 6-8 first). Also skip for exploratory/prototype code.
> **Tools:** mutmut (optional: `pip install mutmut`).

```text
You are analyzing test-suite strength for a Python pipeline using mutation testing.

Goal: Determine whether existing tests detect realistic code changes. Identify blind spots.

Tasks:
1. Identify high-value mutation targets:
   - Comparison operators in validation (== ↔ !=, > ↔ >=, in ↔ not in)
   - Boolean conditions (and/or swaps, negation)
   - Return values (return None, return empty list, return stale default)
   - String operations in slug generation, regex patterns, strip/lower/replace
   - Numeric constants (0 ↔ 1, off-by-one in slicing or counting)
   - Control flow (skip a validation step, remove a break, alter loop bound)
   - Exception handling (remove a raise, broaden except, swallow error)

2. For each target, predict: would current tests catch this mutation?

3. Focus on pipeline-specific semantic mutants:
   - Change a dict key name in metadata extraction
   - Drop one required field from output
   - Alter slug generation rules (separator, casing, truncation)
   - Weaken a validation check (allow empty content through)
   - Swap write-before-validate vs validate-before-write
   - Return "success" after catching an exception

4. Classify each mutant: killed / survived / equivalent.
5. For survived mutants: what test or assertion would kill it?

Practical follow-up (if mutmut is available):

```
mutmut run --paths-to-mutate=<target_module>.py --tests-dir=tests/
mutmut results
```

Output:
- Mutation target map by pipeline stage
- Top 10 most dangerous survived mutants
- Missing assertions to add
- mutmut commands (if applicable)

Rules: Cite exact code locations. Distinguish survived from equivalent mutants.
```

## 10 · Differential & Regression Testing

> **When to use:** After code changes, when multiple modes/flags exist, or when baseline output is available.
> **When to skip:** Greenfield projects with no prior output and no mode flags.
> **Tools:** pytest.

```text
You are designing differential and regression tests for a Python pipeline with multiple modes, skip flags, and caching.

Goal: Detect semantic drift and unintended changes by comparing outputs across configurations or versions.

Comparisons to design:
- Cache-enabled vs cache-disabled (fresh parse): same output?
- All skip flags off vs selective skip flags on: downstream stages still coherent?
- Current branch vs previous known-good output (golden fixtures).
- Strict validation mode vs normal mode: same records pass?
- Full batch vs split-batch-then-merge (if pipeline claims batch independence).

Tasks:
1. Identify all meaningful modes, flags, and behavioral switches in the codebase.
2. Build a replayable corpus: golden fixtures, previously failing inputs, edge cases, empty batches.
3. Define normalization rules for comparing outputs fairly: canonical sort, ignore timestamps, strip non-semantic metadata.
4. Write a comparison harness that runs paired configurations and diffs normalized output.
5. Flag as likely bugs unless explicitly allowed:
   - Missing or extra records between modes
   - Changed field values without documented reason
   - Different exit codes for equivalent inputs
   - Summary counts diverging from row-level data
   - Cache producing different results from fresh processing

For each mismatch, provide:
- Compared modes or versions
- Exact input fixture
- Minimal diff
- Whether the change is intentional, ambiguous, or broken
- Regression test to lock desired behavior

Output: mode/flag comparison matrix, corpus plan, normalization rules, comparison harness, mismatch findings.

Rules: Cite exact flags and modes from the code. No fabricated configurations.
```

## 11 · Logging & Observability Audit

> **When to use:** Production pipelines, CI/CD pipelines, any code where log output is the primary debugging tool.
> **When to skip:** Throwaway scripts or prototypes.
> **Tools:** None.

```text
You are auditing the logging and observability of a Python pipeline to verify that log output is accurate, actionable, and sufficient for diagnosing failures.

Goal: Find misleading log messages, missing error context, false-success claims, and observability gaps.

Focus areas:
- Accuracy: do log messages match actual program state? Does "OK" always mean success?
- Severity consistency: are prefixes like FAIL, WARN, SKIP, OK, INFO used consistently and correctly?
- Error context: when errors are logged, do messages include the input that triggered them, the stage that failed, and enough detail to reproduce?
- False-success claims: final summary messages that claim success when output is empty, partial, or invalid.
- Count accuracy: are logged counts (saved, skipped, failed) measured at the right point — after filtering, not before?
- Sensitive data: API tokens, full response bodies, file paths with usernames, or credentials appearing in error messages.
- Diagnosability: could a developer diagnose any pipeline failure using only the log output, without access to intermediate files?
- Silent failures: code paths that fail without logging anything.
- Log timing: are important state transitions (stage start, stage end, artifact written) logged?

Instructions:
1. Trace every print/log statement. Verify it reports what actually happened.
2. For each finding: severity, confidence, exact location, what the log says vs what actually occurred, fix.
3. After findings: list missing log points and misleading messages to fix.

Be suspicious of:
- Summary lines that count from a variable set before filtering
- "Complete" messages that print regardless of failure count
- Error paths that continue execution after logging FAIL
- Logged filenames that don't match actual written paths

Output: log-statement audit table, misleading messages, missing log points, sensitivity findings.

Rules: Cite exact print/log statements. No fabrication. Mark confidence.
```

## 12 · Idempotence & Recovery Testing

> **When to use:** Pipelines that run on schedules, use caches, or write incremental output.
> **When to skip:** Stateless one-shot scripts with no caching or intermediate files.
> **Tools:** pytest.

```text
You are testing whether a Python pipeline produces consistent results on re-run and recovers safely from interruption.

Goal: Find cases where re-running the pipeline on the same input produces different output, or where stale/partial intermediate state causes corruption.

Focus areas:
- Idempotence: does running the full pipeline twice on identical input produce byte-identical output?
- Cache staleness: if cache files exist from a previous run, does the pipeline validate them or blindly trust them?
- Partial writes: if the pipeline is interrupted mid-write, what state do output files end up in? Does the next run detect and recover?
- Stale intermediates: if L1 files exist but L2 processing is re-run, do old L2 files from deleted L1 sources get cleaned up?
- Atomic writes: are files written atomically (write-to-temp-then-rename) or directly (truncate-then-write, risking corruption)?
- Timestamp sensitivity: does output change based on current time, even when input is identical?
- Directory cleanup: does the pipeline remove previous output before writing, or append/merge?

Tests to write:
1. Run pipeline function twice with same input → assert identical output.
2. Pre-populate cache with stale/mismatched data → assert pipeline detects or overwrites correctly.
3. Pre-populate output directory with leftover files from a previous run → assert no ghost artifacts survive.
4. Simulate partial failure (mock a function to raise mid-batch) → assert no corrupted output is written.
5. Change system clock mock between runs → assert output stability.

Output:
- Idempotence test code
- Recovery scenario tests
- Stale-state risks found in the code
- Recommendations for atomic writes and cleanup

Rules: Cite exact file operations and cache logic. No fabricated paths.
```

## 13 · CI/CD Workflow Audit

> **When to use:** Pipelines that run in CI/CD (GitHub Actions, GitLab CI, etc.).
> **When to skip:** Local-only scripts with no CI integration.
> **Tools:** None.

```text
You are auditing a CI/CD workflow (GitHub Actions YAML) and the Python scripts it orchestrates.

Goal: Find mismatches between the workflow definition and the scripts it calls — wrong paths, missing artifacts, env-var gaps, and failure-handling holes.

Focus areas:
- Artifact flow: do uploaded artifact paths match what scripts actually write? Do download paths match what consuming scripts read?
- Environment variables: are all env vars required by scripts set in the workflow? Are any set but never read, or read but never set?
- Matrix strategy: do parallel matrix jobs write to conflicting paths? Does merge-multiple handle overlapping filenames correctly?
- Dependency consistency: Python version, pip packages, system packages (pandoc, npm, node) — same across all jobs?
- Failure propagation: if an extraction job fails, does the pipeline halt or does the process job run with missing input?
- Skip-flag propagation: do workflow_dispatch inputs actually reach the scripts as expected env vars?
- Deploy safety: can the deploy step run with empty or corrupted staging artifacts? Is there a validation gate?
- Caching: are cache keys specific enough to avoid stale-data bugs? Do restore-keys risk loading incompatible old caches?
- Secret handling: are secrets passed only where needed? Could they leak into logs or artifacts?

Instructions:
1. Cross-reference every env var, path, and artifact name between YAML and Python scripts.
2. Trace the artifact lifecycle: upload → download → read → process → upload → deploy.
3. For each finding: severity, confidence, exact YAML location + Python location, failure scenario, fix.

Output: artifact-flow map, env-var cross-reference, findings table, safety recommendations.

Rules: Cite exact YAML keys and Python references. No fabrication.
```

## 14 · Generated Test Quality Review

> **When to use:** After prompts 6-8 generate test code. Review before trusting.
> **When to skip:** If tests were written by hand and already reviewed.
> **Tools:** None.

```text
You are reviewing AI-generated test code for correctness and effectiveness.

Goal: Verify that generated tests actually test what they claim, will catch real bugs, and won't give false confidence.

Checks to perform:
1. Target existence: does every tested function actually exist in the codebase with that exact name and signature?
2. Import correctness: do imports reference real modules and paths? Will the test file actually run?
3. Assertion strength: are assertions specific enough to catch bugs, or are they tautological (always true)?
   - Flag: `assert result is not None` when the function never returns None.
   - Flag: `assert isinstance(result, str)` without checking the string content.
   - Flag: Tests with no assertions at all.
4. Mock accuracy: do mocks replace external dependencies, or do they accidentally mock the function under test?
5. Edge-case coverage: do tests actually exercise the edge cases they claim to (empty input, None, malformed data)?
6. Independence: can each test run in isolation, or do tests depend on execution order or shared mutable state?
7. False-pass risk: would the test still pass if the function under test were replaced with `return None` or `return ""`?
8. Fixture validity: do test fixtures represent realistic data matching the pipeline's actual data shapes?

For each issue:
- Which test function
- What's wrong
- Why it creates false confidence
- How to fix it

Output: test-by-test review, weak assertions flagged, missing coverage, corrected test code where needed.

Rules: Only review tests that reference real functions. Flag any test targeting fabricated code.
```

## 15 · Finding Aggregation & Remediation Plan

> **When to use:** After running multiple prompts. Final synthesis step.
> **When to skip:** If you only ran one prompt.
> **Tools:** None.

```text
You are consolidating findings from multiple analysis prompts into a single prioritized remediation plan.

Input: findings from previous prompts (provide all findings, or summarize them).

Tasks:
1. Deduplicate: merge findings that describe the same underlying issue from different prompts. Keep the most precise description.
2. Resolve conflicts: if two prompts assigned different severities to the same issue, explain and choose one.
3. Group by risk theme:
   - Silent data corruption / false success
   - Security and trust-boundary violations
   - Missing error handling / dead validation
   - Configuration and schema drift
   - Dead code and unreachable logic
   - Test-suite gaps
   - Logging and observability
   - CI/CD and deployment safety
4. Within each group, rank by: severity (critical first), then confidence (confirmed first), then ease of fix (quick wins first).

   - Phase 2 (short-term): high-severity findings and test-suite gaps.
   - Phase 3 (hardening): medium findings, observability improvements, config cleanup.
   - Phase 4 (maintenance): low findings, dead-code removal, documentation.
6. For each item in the plan: one-line description, original prompt number, severity, confidence, effort estimate (small/medium/large).

Output:
- Deduplicated finding count vs original finding count
- Risk-theme summary table
- Phased remediation plan
- Top 5 highest-impact fixes to make first

Rules: Only reference findings from the provided input. Do not generate new findings. No fabrication.
```
